Phase 1 — Developer Commit to qTest Sequence Flow

GitHub Actions Invocation Pattern

yaml — .github/workflows/ci.yml (product repository)
name: CI Pipeline
on: [push, pull_request]

jobs:
  code-scan:
    uses: org/engineering-skills/.github/workflows/code-scan.yml@v2
    with:
      language: python
      severity-threshold: high
    secrets: inherit

  test-generation:
    needs: code-scan
    uses: org/engineering-skills/.github/workflows/test-generation.yml@v2
    with:
      framework: pytest
      coverage-target: 80

  regression:
    needs: test-generation
    uses: org/engineering-skills/.github/workflows/regression.yml@v2

  qtest-upload:
    needs: regression
    uses: org/engineering-skills/.github/workflows/qtest-upload.yml@v2
    secrets:
      QTEST_TOKEN: ${{ secrets.QTEST_TOKEN }}
04a — Proposed Architecture

CI/CD Platform
Swim Lane Architecture

Each swim lane maps one actor to its responsibilities. The flow runs left-to-right chronologically — from developer trigger through GitHub Actions, Skills Repo checkout, live environment testing, and result delivery to qTest and Webex recipients simultaneously across DEV and QA.

LaneResponsibility
DeveloperTriggers workflow_dispatch or push / PR on any app repo
GitHub ActionsSpins up runner, checks out Skills Repo, installs deps, executes skills
Skills RepoHolds all Playwright suites, API test suites, config, reusable workflow YAML
Live EnvironmentsDEV and QA environments receive HTTPS test requests from the runner
ReportingHTML artifact → Summary → qTest + Webex bot → Recipients

06 — AI Integration

GitHub Copilot & AI
Consumer Architecture

GitHub Copilot, Claude Code, and other AI consumers are first-class clients of the Skills Repository. They invoke skills through the same interfaces as GitHub Actions — making AI a force multiplier that amplifies standardized engineering practices rather than bypassing them.

GitHub Copilot → Skills Repository Invocation Flow
